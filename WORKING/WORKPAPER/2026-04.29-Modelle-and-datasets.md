## 1. Review: Aktueller Projektstand

### Was funktioniert gut

Die bestehende Architektur ist sauber und zielgerichtet. Die Kern-Entscheidungen sind richtig:

- **Vue 3 + Vite**, mobil-first, `<keep-alive>` für Tabs → Performance stimmt
- **`@mlc-ai/web-llm`** als Engine → gut dokumentiert, aktiv gepflegt, funktioniert mit WebGPU
- **Cache-Logik** via `hasModelInCache` → Offline-Nutzung ist korrekt vorbereitet
- **Globaler Ladezustand** mit Progress-Overlay → UX-seitig gut gelöst
- **ELO-Score** (`score`) bereits im State → Arena-Bewertung ist vorbereitet
- **Reaktives State-Management** ohne Pinia/Vuex → für diesen Scope angemessen

### Offene Punkte / technische Schulden

| Bereich | Befund | Priorität |
|---|---|---|
| `state.js` | Kein Fehler-Reset nach GPU-Error | Mittel |
| `App.vue` | Kein PWA-Manifest / Service Worker sichtbar | Hoch (für Offline-Versprechen) |
| Arena | Bewertungslogik noch nicht implementiert | Hoch |
| Modelle | Nur 4 Modelle, alle MLC-Format | Mittel |
| Chat | Kein Streaming sichtbar | Niedrig |

## 2. Analyse der Modellliste

### 2.1 Framework-Realität: Drei verschiedene Ansätze

Das ist der wichtigste technische Punkt dieses Workpapers. Die eingereichte Modellliste enthält Modelle aus **drei verschiedenen Ausführungs-Frameworks**, die **nicht gegenseitig kompatibel** sind:

```
Framework A: @mlc-ai/web-llm       → MLC-kompilierte Modelle (.wasm + Gewichte)   [aktuell genutzt]
Framework B: @huggingface/transformers (transformers.js) → ONNX-Modelle            [webml-community Spaces]
Framework C: wllama                → llama.cpp als WebAssembly, kein GPU nötig      [webcpu Spaces]
```

Jeder Wechsel oder jede Erweiterung erfordert, dass die entsprechende Library geladen und die Initialisierungs-API angepasst wird. Das ist lösbar, aber es ist Arbeit.

## 3. Arena-Bewertungssystem

### 3.1 Philosophie

Das Bewertungssystem soll für jeden verständlich sein – auch für den "dunkelsten anzunehmenden User". Keine abstrakten Zahlen ohne Kontext. Stattdessen: klare, verständliche Kategorien mit Symbolen und kurzen Labels.

### 3.2 Bewertungs-Dimensionen

Jede Arena-Runde bewertet automatisch diese Dimensionen, plus eine manuelle User-Wahl:

| Dimension | Symbol | Was wird gemessen | Automatisch? |
|---|---|---|---|
| **Geschwindigkeit** | ⚡ | Token/Sekunde bis erste Ausgabe | ✅ Ja |
| **Antwortlänge** | 📏 | Tokens gesamt | ✅ Ja |
| **Kohärenz** | 🧠 | Folge-Prompt: "Erkläre deinen letzten Satz" – dann AI selbst prüfen via Mini-LLM | ⚠️ Optional |
| **Deutsch/Multilingual** | 🌍 | Sprachspezifische Testfragen | ✅ Ja (Prompt-Kategorie) |
| **Mathematik** | 🔢 | Rechenaufgaben mit verifizierbarem Ergebnis | ✅ Ja (Prompt-Kategorie) |
| **Kreativität** | 🎨 | Offene Kreativ-Prompts | ✅ Ja (Prompt-Kategorie) |
| **User-Wahl** | 👍 | "Welche Antwort war besser?" | ✅ Kern-Mechanik |

### 3.3 Arena-Ablauf (UX-Spezifikation)

```
1. User wählt Prompt-Kategorie ODER schreibt eigenen Prompt
         ↓
2. Beide Modelle antworten parallel (oder sequenziell wenn RAM knapp)
         ↓
3. Anzeige: Seite A vs. Seite B (Modellnamen erst NACH Abstimmung sichtbar – Blind-Test!)
         ↓
4. User wählt: "A war besser" | "B war besser" | "Unentschieden"
         ↓
5. Reveal: Welches Modell war A/B?
         ↓
6. ELO-Update: Gewinner +K, Verlierer -K (K=32 Standard)
         ↓
7. Bewertungskarte: ⚡ Speed • 📏 Länge • Kategorie-Badge
```

**Blind-Test ist wichtig:** User sollen nicht beeinflusst werden durch "das ist das große Modell" – erst nach der Wahl wird enthüllt.

### 3.5 ELO-Scoring

Das ELO-System ist bereits in `state.js` angelegt (`score: 1200` als Startwert). Empfohlene Implementierung:

```javascript
// Einfache ELO-Berechnung
function updateElo(winnerScore, loserScore, k = 32) {
  const expectedWinner = 1 / (1 + Math.pow(10, (loserScore - winnerScore) / 400));
  const expectedLoser  = 1 - expectedWinner;
  
  return {
    newWinnerScore: Math.round(winnerScore + k * (1 - expectedWinner)),
    newLoserScore:  Math.round(loserScore  + k * (0 - expectedLoser)),
  };
}

// Unentschieden:
function updateEloDraw(scoreA, scoreB, k = 32) {
  const expectedA = 1 / (1 + Math.pow(10, (scoreB - scoreA) / 400));
  return {
    newScoreA: Math.round(scoreA + k * (0.5 - expectedA)),
    newScoreB: Math.round(scoreB + k * (0.5 - (1 - expectedA))),
  };
}
```

ELO-Scores werden in `localStorage` persistiert, damit der Leaderboard über Sessions hinweg wächst.

## 5. Wichtigste Entscheidung: Dual-Framework

Die zentrale Weichenstellung für Phase 2 ist diese:

**Option A – MLC-only bleiben:**  
Nur Modelle einbinden, die im MLC-Format verfügbar sind. Einfach, konsistent, wenig Code. Nachteil: kleinere Modellauswahl.

**Option B – Dual-Engine (MLC + transformers.js):**  
Beide Frameworks parallel nutzen, jedes Modell bekommt ein `engine: 'mlc' | 'transformers'` Flag. Mehr Aufwand, aber deutlich mehr Modelle und Zugang zur gesamten webml-community.

**Empfehlung:** Phase 1 mit MLC-only, Phase 2 mit Dual-Engine. Der Code-Split ist überschaubar: eine `getEngine(model)` Funktion, die je nach Flag die richtige Library initialisiert.
